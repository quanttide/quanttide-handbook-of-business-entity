# 业务需求文档

定义 BRD（业务需求文档）的核心方法和规范。

## 问题

在大多数团队中，需求通常是这样产生的：业务说"我们需要一个XX功能"，产品开始设计方案，开发开始实现，最后发现问题并没有被解决。

根本原因不是执行问题，而是团队从一开始就在"假问题"上做"真努力"。

## 核心改变

BRD 只做一件事：把"需求表达"改造成"问题建模"。

具体改变三件事：

1. 从「要做什么」→「发生了什么」

传统方式：我们需要一个审批系统

BRD方式：在跨部门审批时，发起方需要反复确认责任人，导致流程延迟

从结论回到现场。

2. 从「功能列表」→「业务场景」

传统方式：审批模块、通知模块、权限模块

BRD方式：跨部门审批协同场景、异常处理场景、信息同步场景

系统不再是模块，而是行为的组合。

3. 从「直接给方案」→「延迟决策」

传统方式：需要支持审批流配置

BRD方式：如果存在这样一个平台，能够让责任确认在同一上下文中完成

先把问题讲清，再决定怎么做。

## 场景定义

场景 = 一次完整的业务行为闭环

必须包含：
- 触发（为什么开始）
- 过程（发生了什么）
- 结果（产生了什么）

错误理解：一个按钮点击（太小）、一个公司级流程（太大）

正确认知：跨部门审批一次请求的完整过程。

## 标准结构

每一个场景必须符合：

```
## 场景 X：<名称>

### 工作角色
（谁在参与，以及各自做什么）

### 问题
（业务是如何"卡住"的）

### 假设
如果存在这样一个平台，能够……
```

## 如何写"问题"

正确的问题必须回答四件事：
1. 在什么情况下发生（触发）
2. 现在是怎么做的（现状）
3. 卡在哪里（困难）
4. 导致了什么（后果）

示例：

错误写法：流程效率低

正确写法：在跨部门审批时（触发），发起方需要通过多个沟通工具逐一联系审核方（现状），过程中经常出现责任不清或信息不同步（困难），导致审批周期延长至2-3天（后果）。

## 边界

BRD 的边界是：只描述"问题"和"能力变化"，不描述"功能"。

错误写法：假设支持审批流配置和自动通知

正确写法：如果存在这样一个平台，能够让所有参与方在同一上下文中完成责任确认与反馈。

## 角色定义

角色不是岗位，而是：在这个场景中承担某种行为的人。

错误：产品经理、财务、CEO

正确：发起方、审核方、执行方

## 本质

BRD 不是"写需求"，而是：把模糊的业务困扰，转化为可验证的问题结构。

## 自检

如果一份 BRD 同时满足以下条件，才算合格：
- 能让一个完全不了解业务的人"看见现场"
- 不包含任何具体功能描述
- 不同的人读完，对问题理解一致
- 可以被 QA 验证"问题是否被解决"

## 检验

如果写完 BRD 后：
- 很自然就能开始画功能结构
- 不同人开始提出不同方案
- 但大家对问题没有争议

说明这份 BRD 是成功的。否则，大概率只是写了一份"换皮的 PRD"。
