# 业务需求文档

定义 BRD（业务需求文档）的核心方法和规范。

## 本质

### 问题

在大多数团队中，需求是这样产生的：业务说"我们需要一个XX功能"，产品设计方案，开发实现，最后发现问题没被解决。

根本原因不是执行问题，而是团队从一开始就在"假问题"上做"真努力"。

### 核心改变

BRD 只做一件事：把"需求表达"改造成"问题建模"。

1. 从「要做什么」→「发生了什么」：从结论回到现场
2. 从「功能列表」→「业务场景」：系统不再是模块，而是行为的组合
3. 从「直接给方案」→「延迟决策」：先把问题讲清，再决定怎么做

### 边界

BRD 只描述"问题"和"能力变化"，不描述"功能"。

错误：假设支持审批流配置和自动通知

正确：如果存在这样一个平台，能够让所有参与方在同一上下文中完成责任确认与反馈。

### 自检

合格的 BRD 满足：
- 能让不了解业务的人"看见现场"
- 不包含具体功能描述
- 不同人读完，对问题理解一致
- 可被 QA 验证"问题是否被解决"

### 检验

写完 BRD 后：
- 很自然就能开始画功能结构
- 不同人提出不同方案
- 但大家对问题没有争议

说明 BRD 是成功的。否则，只是写了一份"换皮的 PRD"。

## 方法

### 场景定义

场景 = 一次完整的业务行为闭环，必须包含：触发、过程、结果。

错误理解：一个按钮点击（太小）、一个公司级流程（太大）

正确认知：跨部门审批一次请求的完整过程。

### 标准结构

```
## 场景 X：<名称>

### 工作角色
（谁在参与，以及各自做什么）

### 问题
（业务是如何"卡住"的）

### 假设
如果存在这样一个平台，能够……
```

### 如何写"问题"

正确的问题必须回答四件事：触发、现状、困难、后果。

画面感：描述真实的业务困扰。

反例：数据同步效率低，存在信息孤岛。

正例：销售在客户现场签单时，需要反复打电话给仓库确认库存，导致单笔订单成交时间延长了 20 分钟，且经常出现"下单后发现无货"的尴尬。

### 假设句式

使用「如果存在这样一个平台，能够...」

这是 BRD（问题空间）到 PRD（方案空间）的唯一接口。它既定义了需求边界，又给产品经理留下创意空间。

### 角色定义

角色不是岗位，而是：在这个场景中承担某种行为的人。

错误：产品经理、财务、CEO

正确：发起方、审核方、执行方

#### 角色 vs 岗位

使用工作角色（如发起方、确认方），而非具体岗位。

原因：岗位会随组织架构调整而变动，但角色在业务流中是稳固的。这能保证 BRD 的生命周期更长。
